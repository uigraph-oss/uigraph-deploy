# UiGraph on AWS EKS

A production-grade path for self-hosting UiGraph on an existing AWS EKS cluster: a Helm chart
that runs the four stateless app services (`uigraph-api`, `uigraph-graphql`, `uigraph-gateway`,
`uigraph-ui`) as Kubernetes Deployments, backed by managed AWS services instead of the bundled
Postgres/Redis/MinIO used by the root [`docker-compose.yml`](../docker-compose.yml).

This tree does **not** create a VPC, EKS cluster, or node groups — that's assumed to already
exist. It provisions the *data plane* (RDS, ElastiCache, S3, IAM) with Terraform and the *app*
with Helm.

```
k8s/
├── helm/uigraph/     # Helm chart — the four app Deployments, Services, Ingress, HPA, PDB
└── terraform/         # RDS (Postgres), ElastiCache (Redis), S3, IRSA IAM role
```

## Prerequisites

1. **An existing EKS cluster** with an IAM OIDC provider associated (required for IRSA):
   ```bash
   aws eks describe-cluster --name <cluster-name> --query "cluster.identity.oidc.issuer" --output text
   # if not yet associated:
   eksctl utils associate-iam-oidc-provider --cluster <cluster-name> --approve
   ```
2. **AWS Load Balancer Controller** installed in the cluster (this chart's Ingress assumes
   `ingressClassName: alb`):
   ```bash
   helm repo add eks https://aws.github.io/eks-charts
   helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
     -n kube-system --set clusterName=<cluster-name>
   ```
   See the [AWS Load Balancer Controller docs](https://kubernetes-sigs.github.io/aws-load-balancer-controller/)
   for the IAM policy/IRSA role it itself needs.
3. **An ACM certificate** (in the same region as the ALB) covering your app and gateway
   hostnames, e.g. `app.example.com` and `sync.example.com`, or a wildcard.
4. CLI tools: `kubectl` (configured against the cluster), `helm` >= 3.12, `terraform` >= 1.5,
   `aws` CLI authenticated to the target account.
5. Note down: the cluster's VPC ID, at least two private subnet IDs, the OIDC provider ARN and
   issuer URL (from step 1), and the security group ID shared by EKS worker nodes/pods (the
   cluster's primary security group, or your node groups' shared SG).

## 1. Provision the data plane (Terraform)

```bash
cd k8s/terraform
cp terraform.tfvars.example terraform.tfvars
# edit terraform.tfvars: vpc_id, subnet_ids, eks_oidc_provider_arn, eks_oidc_provider_url,
# eks_node_security_group_id, bucket_name (must be globally unique), aws_region

terraform init
terraform plan
terraform apply
```

This creates:
- An RDS Postgres instance. The master password is generated by AWS and stored directly in
  Secrets Manager (`manage_master_user_password`) — it's never in Terraform state or output.
- An ElastiCache Redis replication group (TLS-in-transit always on; AUTH token optional via
  `redis_auth_token_enabled`/`redis_auth_token`).
- An S3 bucket (versioned, SSE-encrypted, no public access) for UiGraph's object storage.
- An IAM role trust-scoped to the `uigraph` ServiceAccount (IRSA) with least-privilege access to
  just that bucket.

Note the outputs — you'll need them for the Helm install:

```bash
terraform output
```

## 2. Create the app Secret

The chart needs `UIGRAPH_SECRET_KEY`, `STORAGE_ACCESS_KEY`/`STORAGE_SECRET_KEY`, the Postgres
password, and `UIGRAPH_ADMIN_PASSWORD` in a Kubernetes Secret. Two options:

**Option A — recommended for production: External Secrets Operator**, syncing straight from the
Secrets Manager entry Terraform created for the RDS password, plus whatever you use for the
others:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: uigraph
spec:
  secretStoreRef: { name: aws-secrets-manager, kind: ClusterSecretStore }
  target: { name: uigraph-external-secret }
  data:
    - secretKey: postgres-password
      remoteRef: { key: <postgres_master_user_secret_arn output>, property: password }
    # ...uigraph-secret-key, storage-access-key, storage-secret-key, admin-password similarly
```

Then set `secrets.existingSecret: uigraph-external-secret` in your Helm values and skip
`secrets.values` entirely.

**Option B — manual, for a quick eval install**:

```bash
kubectl create secret generic uigraph \
  --from-literal=uigraph-secret-key="$(openssl rand -hex 32)" \
  --from-literal=storage-access-key="<IAM user access key, if not using IRSA>" \
  --from-literal=storage-secret-key="<IAM user secret key, if not using IRSA>" \
  --from-literal=postgres-password="<from Secrets Manager: terraform output postgres_master_user_secret_arn>" \
  --from-literal=admin-password="$(openssl rand -hex 16)"
```
Then set `secrets.existingSecret: uigraph` (the chart's default key names match this command).

## 3. Install the Helm chart

At minimum, override:

```bash
helm install uigraph k8s/helm/uigraph \
  --set postgres.host=<terraform output postgres_endpoint> \
  --set redis.host=<terraform output redis_primary_endpoint> \
  --set storage.bucket=<terraform output s3_bucket_name> \
  --set storage.publicEndpoint=<terraform output s3_bucket_regional_domain_name> \
  --set app.domain=example.com \
  --set ingress.certificateArn=<your ACM certificate ARN> \
  --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=<terraform output irsa_role_arn> \
  --set secrets.existingSecret=uigraph
```

Or write these into a `values-prod.yaml` and pass `-f values-prod.yaml`. See
[`helm/uigraph/values.yaml`](helm/uigraph/values.yaml) for every available setting and its
default.

Watch it come up:

```bash
kubectl get pods -l app.kubernetes.io/instance=uigraph -w
kubectl get ingress uigraph
```

Once the ALB is provisioned (can take a few minutes), point DNS (`app.<domain>` and, if the
gateway ingress is enabled, `sync.<domain>`) at its hostname. Sign in at `https://app.<domain>`
with `UIGRAPH_ADMIN_EMAIL` / the password you put in the Secret.

### IRSA vs. static IAM-user credentials

The chart's `serviceAccount` is annotated with the IRSA role ARN by default, but
`STORAGE_ACCESS_KEY`/`STORAGE_SECRET_KEY` are still required, non-empty config for
`uigraph-api`/`uigraph-gateway` today (verify against those services' current source before
assuming otherwise) — so the safe default is to also populate them with a static IAM user's
keys. If a future `uigraph-api` release falls back to the default AWS credential chain when
those two values are blank, IRSA alone is sufficient and the static keys can be dropped; until
then, treat IRSA as defense-in-depth, not the sole credential path.

## Upgrades

```bash
helm upgrade uigraph k8s/helm/uigraph -f values-prod.yaml
```

Bump `api.image.tag` / `graphql.image.tag` / `gateway.image.tag` / `ui.image.tag` (or pass
`--set`) to roll to a new UiGraph release. `uigraph-api` applies its own migrations on startup,
same as in `docker-compose.yml` — no separate migration step is needed.

## Backup / restore

- **Postgres**: RDS automated backups are enabled with a 7-day retention window
  (`backup_retention_period` in `k8s/terraform/rds.tf`); `deletion_protection` is on. Take a
  manual snapshot before major upgrades: `aws rds create-db-snapshot ...`.
- **Object storage**: the S3 bucket has versioning enabled, so overwritten/deleted objects are
  recoverable via `aws s3api list-object-versions`.
- **Redis**: used as a cache/queue, not treated as a durable store — no automated ElastiCache
  snapshotting is configured. Add `snapshot_retention_limit` to
  `aws_elasticache_replication_group.uigraph` if you need it.

## Troubleshooting

- **ALB never provisions / Ingress has no address**: check the AWS Load Balancer Controller logs
  (`kubectl logs -n kube-system deploy/aws-load-balancer-controller`). Common causes: the
  controller isn't installed, its IRSA role lacks permissions, or `ingress.certificateArn` is in
  a different region than the cluster.
- **502/504 from the ALB**: target group health checks are failing. Check
  `kubectl get pods` for the relevant service and `kubectl logs` — the ALB controller derives
  target health from the same `/healthz` (or `/` for `uigraph-ui`) checks as the pod's readiness
  probe.
- **`uigraph-api` CrashLoopBackOff on startup**: almost always the Postgres connection. Confirm
  `postgres.host`/`sslmode` are correct and that the RDS security group
  (`aws_security_group.postgres` in Terraform) allows ingress from the EKS node/pod security
  group you passed as `eks_node_security_group_id`.
- **Figma OAuth redirect fails**: `FIGMA_REDIRECT_URI` is derived from `app.domain`
  (`https://app.<domain>/api/v1/figma/callback` by default) — it must exactly match what's
  registered in the Figma app's OAuth settings.
- **S3 access denied from `uigraph-api`/`uigraph-gateway`**: check both the IRSA trust policy
  (`k8s_namespace`/`k8s_service_account_name` in `terraform.tfvars` must match where the chart
  is actually installed) and, if not relying on IRSA yet, that `STORAGE_ACCESS_KEY`/
  `STORAGE_SECRET_KEY` in the Secret are a valid, non-expired IAM user key pair.

## Known gaps / assumptions

- **Redis TLS+AUTH**: whether `uigraph-api`'s `REDIS_URL` parsing supports `rediss://` and an
  inline AUTH token hasn't been confirmed against that service's source. The chart defaults to
  plain `redis://` (`redis.tls: false`, `redis.authEnabled: false`) — no TLS, no AUTH — which is
  a real security gap versus RDS (which does get `sslmode=require` by default). If you confirm
  `uigraph-api` supports it, set `redis.tls: true` / `redis.authEnabled: true` and
  `redis_auth_token_enabled = true` in Terraform to close this gap.
- **`uigraph-ui` health check**: it has no documented `/healthz` endpoint, so its liveness/
  readiness probes target `/` (the SPA shell) instead.
- **IRSA fallback**: see "IRSA vs. static IAM-user credentials" above — static keys are the
  default until `uigraph-api`'s S3 client behavior with the default AWS credential chain is
  confirmed.

## Uninstall

```bash
helm uninstall uigraph
cd k8s/terraform && terraform destroy   # WARNING: destroys RDS/ElastiCache/S3 — take snapshots first
```
